# 明日方舟气质 UI 设计规划

更新日期：2026-04-10

## 1. 目标

为当前技术博客建立一套可执行的深色轻工业视觉方案，吸收明日方舟官网在“组织档案感、工业导视感、冷静层级感”上的长处，但不把站点做成重运营、重活动页式的游戏官网。

本规划的目标不是视觉模仿，而是把可复用的设计语言拆成设计 token、组件规则、页面结构和实施阶段，后续可以直接进入实现。

## 2. 设计结论

### 2.1 从明日方舟官网提炼出的可迁移特征

基于 2026-04-10 可访问版本的 `https://ak.hypergryph.com/` 观察，风格核心可以拆为以下几点：

1. 深底不是纯黑炫技，而是为了服务“档案系统 / 组织终端 / 情报看板”叙事。
2. UI 主体颜色极克制，常见基础色集中在黑、白、深灰、冷灰，鲜艳颜色只做少量高亮。
3. 排版高度依赖英文工业字、小字注释、编号、分隔线、横向导视，不靠拟物或玻璃感。
4. 模块是切片式结构，像一块块独立面板堆叠，而不是圆角卡片瀑布流。
5. 动效以滑入、淡入、轻视差、状态切换为主，强调“系统反馈”，不强调热闹。
6. 角色图承担气氛，UI 本体保持克制。

### 2.2 不应直接照搬的部分

以下做法不适合当前博客：

1. 全站统一纯黑背景和低对比正文。
2. 首页大视频背景、轮播新闻、重营销 Banner。
3. 大量活动 KV、全屏角色立绘、过强动态效果。
4. 明显的 IP 视觉符号挪用，例如罗德岛标记、编号格式、品牌文字结构。

## 3. 站点落地方向

### 3.1 总体方向

推荐路线：

`工业档案式外壳 + 高可读正文内核`

含义如下：

1. 首页、列表页、联系页可以显著转向深色、结构化、导视化。
2. 博文正文页保持长时间阅读舒适度，不做过黑、过小、过密。
3. 页面之间用同一套 token、边框、字阶、信息标签体系串起来。
4. 整站气质偏“维护中的技术档案库”，不是“游戏宣传站”。

### 3.2 当前站点基础

现状里已经有可继承的基础：

1. `src/styles/global.css` 已经有栅格背景、低饱和颜色、`shell-card` 容器和 IBM Plex 字体体系。
2. `src/pages/index.astro` 的首页结构比较克制，适合升级成“档案首页”而不是推倒重来。
3. 导航、卡片、页脚都已经是平面化组件，能较顺滑切到新方案。

因此本次 UI 改造不建议重建信息架构，重点是替换视觉系统和局部版式逻辑。

## 4. 设计原则

### 4.1 视觉原则

1. 深色但不死黑，层次依赖煤灰、冷灰和浅灰。
2. 锐利但不硬核，避免过多斜切和高噪点。
3. 工业但不赛博，避免霓虹蓝紫和过度发光。
4. 克制但不单调，依赖排版、边框、留白和层级制造气质。
5. 中文优先，英文只作为 meta 信息和结构强化。

### 4.2 内容原则

1. 博客首页仍然只展示新内容和必要动作。
2. 不把旧站归档重新塞回首页。
3. 联系入口必须保持清晰可见。
4. 文章列表和标签索引要比装饰更优先。

## 5. 视觉系统

### 5.1 色板建议

#### 基础背景

- `--bg-root: #0d1013`
- `--bg-elevated: #14191d`
- `--bg-panel: rgba(21, 27, 31, 0.92)`
- `--bg-panel-strong: rgba(25, 31, 36, 0.98)`

#### 线框与分隔

- `--line-soft: rgba(226, 232, 238, 0.08)`
- `--line: rgba(226, 232, 238, 0.14)`
- `--line-strong: rgba(226, 232, 238, 0.22)`

#### 文字

- `--text: #edf1f4`
- `--text-muted: #9ea8b2`
- `--heading: #ffffff`
- `--text-faint: #717b84`

#### 强调色

- `--accent-cyan: #77c7d7`
- `--accent-amber: #c8a96b`
- `--accent-red: #b5685d`

使用规则：

1. 整站同时只保留一个主强调色，推荐 `accent-cyan`。
2. `accent-amber` 只用于次级 badge、时间线或编号。
3. `accent-red` 只用于告警、风险、停用状态。

### 5.2 背景语言

背景不做大纹理，应使用以下组合：

1. 超低对比大面积纵向渐变。
2. 局部 radial glow，颜色极淡。
3. 稀疏网格线或工程图式细线。
4. 个别页面可增加扫描线感噪点，但透明度必须极低。

### 5.3 字体策略

#### 正文与中文 UI

继续使用当前的：

- `IBM Plex Sans`
- `Source Serif 4`
- `IBM Plex Mono`

#### 工业 meta 字

新增一套英文工业标题字，仅用于编号、辅助标题、模块标识、按钮英文副标题：

- 首选：`Oswald`
- 备选：`Barlow Condensed`
- 备选：`Rajdhani`

使用限制：

1. 不用于大段中文正文。
2. 不用于长篇文章标题主文案。
3. 只用于 10% 到 15% 的文字面积。

### 5.4 圆角、边框、阴影

#### 圆角

- 全站主圆角下调到 `12px` 至 `16px`
- 小按钮与标签圆角控制在 `8px` 左右

#### 边框

1. 以 1px 细边框为主。
2. 用边框和分隔线表达层级，不用厚投影堆氛围。
3. 局部可加上边/左边强调线，但不要形成双层无意义边框。

#### 阴影

1. 阴影极轻，只表达浮层关系。
2. 不要玻璃拟物，不要软萌阴影。

## 6. 组件级设计规范

### 6.1 Header

对应文件：

- `src/components/SiteHeader.astro`
- `src/styles/global.css`

改造目标：

1. Header 从“浅底圆角导航条”转为“深色终端条”。
2. Brand 区增加副标题编号信息，例如 `NODE // KNOWLEDGE INDEX` 这类抽象 meta 字，但不直接引用游戏文案。
3. 当前导航高亮从整块纯黑反白，改为细边框 + 浅底高亮条 + 小型状态标记。
4. Sticky 保留，但压低视觉重量。

验收标准：

1. 首屏不压迫正文。
2. 当前页状态清晰。
3. 深底下导航对比度不低于 AA 级可读。

### 6.2 首页 Hero

对应文件：

- `src/pages/index.astro`
- `src/styles/global.css`

改造目标：

1. 首页首屏从普通欢迎卡片升级为“档案首页面板”。
2. 左侧保留主标题和说明，右侧可引入一块视觉辅助区，例如抽象网格、编号、短句或角色轮廓占位，但不强依赖图像。
3. CTA 保持少量明确按钮，不做轮播。
4. 增加一层 meta 信息，例如站点定位、更新时间、内容域标签。

结构建议：

1. Eyebrow：站点编号 + 类型标签
2. H1：一句强陈述标题
3. Lead：中文说明
4. Meta Row：主题域、维护状态、内容类型
5. CTA Row：博客 / RSS / 联系

### 6.3 Post Card

对应文件：

- `src/components/PostCard.astro`
- `src/styles/global.css`

改造目标：

1. 卡片结构更像档案条目，而不是普通博客卡片。
2. 时间、难度、阅读时长改造成统一 meta 条。
3. 标题下方可以加入细小英文分类或序号栏。
4. hover 反馈用边框亮起、标题位移、角标变化，不用大面积抬升。

### 6.4 Tag / Button / Filter

涉及文件：

- `src/components/TagChip.astro`
- `src/components/OldBlogFilterSelect.vue`
- `src/styles/global.css`

规范：

1. Tag 采用低厚度扁平样式。
2. Button 改为矩形偏扁、细边框、清晰状态切换。
3. 筛选器遵守统一的面板和选中态规则，不使用默认原生视觉。

### 6.5 Footer

对应文件：

- `src/components/SiteFooter.astro`
- `src/styles/global.css`

改造目标：

1. Footer 设计成“系统尾注区”。
2. 联系入口保留高可见度。
3. 加入简短的维护状态或站点定位，不加多余装饰。

## 7. 页面级实施策略

### 7.1 第一阶段必须覆盖的页面

1. 首页 `/`
2. 博客列表 `/blog/`
3. 博文详情 `/blog/[...slug]`
4. 标签页 `/tags/`
5. 联系页 `/contact/`

### 7.2 第二阶段覆盖页面

1. 路线页 `/routes/`
2. 项目页 `/projects/`
3. 归档页 `/archive/`

### 7.3 `old-blog` 特殊说明

`old-blog` 可以使用同一套深色 token，但不应做成高度风格化页面。优先保持筛选和图表清晰。

## 8. 技术落点

### 8.1 需要修改的核心文件

1. `src/styles/global.css`
2. `src/layouts/BaseLayout.astro`
3. `src/components/SiteHeader.astro`
4. `src/components/SiteFooter.astro`
5. `src/components/PostCard.astro`
6. `src/components/SectionHeading.astro`
7. `src/pages/index.astro`
8. 必要时补充新的装饰组件，例如 `src/components/TechMetaStrip.astro`

### 8.2 推荐实施顺序

1. 先建立新 token 和字体引入。
2. 再改 header、footer、button、tag 等全局基础组件。
3. 再重做首页 hero 和文章卡片。
4. 再收口到列表页、详情页和 `old-blog`。

## 9. 分阶段执行计划

### Phase 1：Token 与基础组件

目标：

1. 完成深色主题变量。
2. 完成 header / footer / button / tag 的统一风格迁移。

产出：

1. 深色变量集
2. 英文 meta 字引入
3. 基础交互态规范

### Phase 2：首页与博客列表

目标：

1. 把首页改造成档案式入口页。
2. 统一文章卡片和分区标题。

产出：

1. 新首页 hero
2. 新 post card
3. 新 section heading

### Phase 3：详情页与索引页收口

目标：

1. 正文页保持阅读舒适前提下完成视觉统一。
2. 标签页、联系页、归档页进入同一设计语言。

产出：

1. 正文页标题区和元信息区更新
2. 联系页增强
3. 各索引页统一完成

## 10. 验收标准

### 10.1 视觉验收

1. 整站在桌面端和移动端都保持统一的深色轻工业气质。
2. 不出现过度赛博、过度玻璃、过度二次元的样式失控。
3. 首屏可辨识，但不牺牲信息入口。

### 10.2 内容验收

1. 首页仍只承接新站内容。
2. 联系入口在首页、联系页、页脚都可见。
3. 博文正文阅读舒适度没有明显下降。

### 10.3 工程验收

1. `npm run build` 通过。
2. 关键页面无明显布局抖动。
3. 深色模式不依赖客户端切换，默认即为设计方案。

## 11. 风险与约束

1. 如果正文页也完全黑化，长文阅读疲劳会明显上升。
2. 如果英文工业字体用得过多，中文页面会显得刻意。
3. 如果角色和装饰元素过重，博客会失去技术站可信度。
4. 当前仓库没有设计 token 分层文件，首次改造时要控制 CSS 扩散。

## 12. 推荐后续动作

1. 先做一版首页和 header 的深色视觉样稿。
2. 再根据样稿落一版 token 和基础组件。
3. 等 UI 外壳稳定后，再接入右下角悬浮角色挂件。
