# 右下角悬浮看板娘挂件技术规划

更新日期：2026-04-10

## 1. 目标

为站点规划一个右下角悬浮的二次元角色挂件，支持待机、眨眼、呼吸、鼠标视线跟随、点击互动和可关闭/折叠，不嵌入正文流，不影响阅读主路径。

本规划优先考虑：

1. 可维护性
2. 页面性能
3. 交互克制
4. 资产和授权可控

## 2. 产品定义

### 2.1 定位

角色挂件不是首页主内容，也不是 AI 聊天产品。它的定位更像：

`全站级悬浮站点助手 / 值班记录员`

### 2.2 必须满足的交互前提

1. 固定在桌面端右下角悬浮。
2. 不进入正文布局，不挤占文章内容宽度。
3. 鼠标移动时，角色眼球必须能跟随指针。
4. 角色头部可以轻微跟随，但幅度要受控。
5. 支持点击头部或身体触发动作或短提示。
6. 用户可以收起、静音或关闭。
7. 移动端默认折叠或禁用，避免遮挡。

### 2.3 明确不做的内容

第一阶段不做：

1. AI 对话
2. 自动语音播报
3. 长时间占据屏幕的大气泡
4. 强打断式通知
5. 页面首次加载时的大型入场动画

## 3. 技术结论

### 3.1 推荐技术路线

推荐采用：

`Live2D 模型 + Cubism SDK for Web 能力模型 + PixiJS 渲染封装`

更具体地说：

1. 资产层：Live2D Cubism 模型
2. 运行时：Live2D Cubism SDK for Web
3. 集成层：优先评估 `pixi-live2d-display`
4. 前端封装：Astro 页面挂载 + Vue 交互控制组件

### 3.2 为什么不优先选 3D VRM

虽然 `three-vrm` 技术成熟，但不适合本阶段：

1. 运行时更重。
2. 对博客场景来说存在感太强。
3. 更接近 VTuber 头像，而不是轻量悬浮助手。
4. 角色风格更容易偏离当前站点的轻工业气质。

### 3.3 为什么不建议直接用“老式看板娘一键脚本”

这类脚本适合快速原型，不适合长期主方案：

1. 可维护性差。
2. 交互和资源管理粒度有限。
3. 很多项目维护状态不稳定。
4. 样式和生命周期很难与当前 Astro 站统一。

## 4. 方案对比

### 4.1 方案 A：官方 Live2D SDK 直接集成

优点：

1. 能力最完整。
2. 可精确控制参数、动作、命中区域和事件。
3. 后续升级空间最大。

缺点：

1. 集成成本更高。
2. 初次接入时工程细节较多。

适用场景：

1. 后续要做较复杂动作编排。
2. 计划长期维护角色系统。

### 4.2 方案 B：`pixi-live2d-display` 封装集成

优点：

1. 对 Web 集成更友好。
2. 常见交互如自动跟随、点击测试、动作调用更省事。
3. 更适合博客里这种局部挂件场景。

缺点：

1. 仍然依赖模型质量。
2. 对部分高级能力仍需理解底层参数。

适用场景：

1. 第一阶段就要尽快落地一个稳定挂件。
2. 后续仍然希望保留可扩展能力。

### 4.3 方案 C：第三方现成 widget 项目

优点：

1. 上线快。
2. 演示成本低。

缺点：

1. 样式难统一。
2. 可控性一般。
3. 长期维护风险高。

适用场景：

1. 只做一次性原型演示。

### 4.4 最终建议

首选 `方案 B`，保留未来必要时向 `方案 A` 下沉的空间。

## 5. 关键能力拆解

### 5.1 视线跟随

这是你明确要求的核心能力，通常依赖 Live2D 标准参数：

1. `ParamEyeBallX`
2. `ParamEyeBallY`
3. `ParamAngleX`
4. `ParamAngleY`
5. 必要时加入轻微 `ParamBodyAngleX`

实现原则：

1. 眼球跟随幅度可以稍大。
2. 头部跟随幅度必须小于眼球。
3. 身体跟随只做极弱补偿。
4. 鼠标离开窗口后平滑回正。

### 5.2 待机动画

建议配置：

1. 呼吸
2. 眨眼
3. 轻微头部浮动
4. 空闲 20 到 40 秒后触发一个低干扰 idle motion

### 5.3 点击互动

建议只保留有限命中区域：

1. 头部
2. 身体
3. 控制按钮区域

触发内容：

1. 播放一段短 motion
2. 显示一条短气泡文本
3. 记录冷却时间，避免连续刷屏

### 5.4 折叠与关闭

需要三种状态：

1. 展开
2. 收起为小按钮或头像徽标
3. 本次访问关闭

状态建议落在 `localStorage` 中持久化。

## 6. 资产要求

### 6.1 模型要求

要实现自然的“看着鼠标”，模型至少要满足：

1. 有完整眼球参数
2. 有头部角度参数
3. 最好有呼吸参数
4. 最好有基础表情和 2 到 4 个 motion
5. 有明确 hit area 配置

### 6.2 资产来源原则

必须满足以下要求：

1. 角色授权允许上站公开展示。
2. 运行时授权允许 Web 集成。
3. 不直接使用明日方舟或其他现成 IP 角色。
4. 最稳的是原创角色或明确可商用授权素材。

### 6.3 建议角色设定

为了与博客气质匹配，角色形象建议是：

1. 偏记录员、技术助理、档案管理员
2. 服装和配色低饱和
3. 不要过度萌系、亮色、糖果感
4. 表情偏冷静、温和、轻陪伴

### 6.4 Live2D 模型获取渠道调研

如果目标是“有较多角色可选的 Live2D hub”，优先级建议如下：

1. `nizima`：Live2D 官方相关的作品交易市场，适合找可购买的 Live2D 模型、插画、素材，也可以按用途发起定制需求。优点是和 Live2D 生态绑定更紧，授权信息相对规范；缺点是每个模型仍需逐条确认可否用于公开 Web 站点、是否允许改造、是否允许二次分发资源文件。
2. `BOOTH`：日本创作者市场，Live2D 相关条目数量很多，适合找现成 VTuber 模型、练习模型、PSD/rigging 素材。优点是量大；缺点是授权完全依赖单个作者的商品说明，常见限制包括禁止再分发、禁止商用、禁止改造、仅允许直播用途。
3. `VGen`：更适合找画师或 rigging 作者做定制委托，而不是直接下载大量现成模型。优点是可以定制原创角色和授权条款；缺点是周期和成本高于现成素材。
4. `Live2D 官方 Sample Data`：适合技术验证、学习和原型。优点是授权边界比随便找网盘素材更清楚；缺点是角色辨识度固定，不适合作为长期站点形象。
5. `itch.io`、`Gumroad` 等泛创作者市场：可以作为补充来源，但需要逐条读 license，不建议作为第一选择。

采购或下载前必须确认这些项：

1. 是否包含 Web 运行所需文件，例如 `.model3.json`、`.moc3`、纹理、motion、expression、physics、hit area。
2. 是否允许部署到公开网站。
3. 是否允许修改尺寸、贴图、动作和文案。
4. 是否禁止把模型资源文件放到公开静态目录。
5. 是否限制用途为直播、视频、头像软件，而不是网页挂件。
6. 是否允许非商用公开展示。非商用不等于自动授权。

### 6.5 自己做一个是否现实

结论：

可以自己做，但建议只把第一版目标定为“简单 chibi/半身助手”，不要一开始做复杂全身 Live2D。

推荐路线：

1. 先画或委托一张分层 PSD。
2. 用 Live2D Cubism Editor 进行切图、网格、变形器、参数和物理设置。
3. 先使用模板功能或官方 Sample Data 学习眼球、头部、呼吸和眨眼参数。
4. 导出 Web 运行所需模型资源。
5. 在站点里用占位引擎验证加载、鼠标跟随、点击动作和折叠逻辑。

可行性判断：

1. 如果只做右下角小助手，chibi 模型比精细半身模型更适合。
2. 如果目标是“鼠标视线跟随”，最重要的不是画得多复杂，而是眼球、头部和呼吸参数是否做完整。
3. Live2D Cubism Editor 的免费版可以做基础模型，但参数、ArtMesh、纹理等有功能限制；复杂模型更适合使用 PRO 试用或正式版。
4. 自己做的最大优势是版权清晰，且可以贴合本站“技术助理 / 档案记录员”的气质。

### 6.6 使用明日方舟或 PRTS 模型的可行性

结论：

不建议把明日方舟现成模型作为本站公开部署方案，即使本站定位是非商用、纯玩。

调研观察：

1. PRTS 角色页确实有“干员模型”区域。
2. 以阿米娅页面为例，页面源码中出现 `SPINEDATA` 和 `spine-root`，且容器带有 `data-id="char_002_amiya"`。这说明 PRTS 页面里的“小人模型”更接近游戏内 Spine 模型展示，不是可直接接入 Live2D Cubism Web Runtime 的 `.moc3` 模型。
3. 因此即使不讨论授权，也不能简单把 PRTS 小人模型当作 Live2D 模型接入 `pixi-live2d-display`。如果要复用这类模型，技术路线会变成 Spine runtime，而不是本规划当前选择的 Live2D runtime。
4. PRTS 的版权页覆盖的是 Wiki 内容和其页面协议，不等于明日方舟游戏资产被重新授权给下游网站使用。
5. 目前没有在本轮调研中找到鹰角明确授权第三方公开网站直接抽取并部署干员模型资源的官方说明。

风险判断：

1. 个人本地实验和公开部署到 GitHub Pages 是两个风险等级。
2. “非商用”只能降低商业冲突感，不能自动解决版权、商标、角色形象和游戏资产再分发问题。
3. 如果把模型文件放到 `public/mascot/` 并部署到 GitHub Pages，本质上是在公开再分发资源文件，风险明显高于只在 PRTS 页面上观看。
4. 如果未来只是本地私玩 demo，可以单独建本地实验分支或本地脚本，但不要进入主站生产路径，也不要 commit 游戏资源文件。

可替代方案：

1. 委托原创角色，但在设定上参考“低饱和、档案员、工业制服、终端助手”的气质。
2. 使用授权清楚的通用 Live2D 模型先做交互验证。
3. 如果一定喜欢明日方舟小人那种“战斗 chibi”质感，可以考虑原创 chibi + Spine 技术路线，但这会偏离当前 Live2D 方案。

### 6.7 资产路线建议

当前推荐顺序：

1. 第一选择：原创或委托的 Live2D chibi/半身模型。
2. 第二选择：从 `nizima` 或 `BOOTH` 购买授权明确、允许 Web 展示的 Live2D 模型。
3. 第三选择：官方 Sample Data 做本地原型，但不作为长期角色。
4. 不推荐：使用明日方舟、PRTS 或其他游戏 IP 的提取模型作为公开站点资源。

### 6.8 按技术格式划分的模型来源

如果只从“技术上哪里能拿到可用模型”来划分，当前常见来源可以分成下面四类：

#### A. 现代 Live2D Cubism 模型

典型文件结构：

1. `*.model3.json`
2. `*.moc3`
3. `*.motion3.json`
4. `*.exp3.json`
5. `*.physics3.json`
6. `*.pose3.json`
7. `*.cdi3.json`
8. 纹理图集文件

这类模型是当前方案最理想的来源，能直接进入 `pixi-live2d-display + Cubism 4 Core` 路线。

主要获取侧：

1. Live2D 官方 Sample Data 下载页
2. `nizima`
3. `BOOTH`
4. 作者个人发布页或 GitHub 仓库

耳熟能详且技术上明确属于这一路的角色示例：

1. `Haru`
2. `Haru (receptionist version)`
3. `Hiyori Momose`
4. `Niziiro Mao`
5. `Hatsune Miku`
6. `Unity-chan`

风险备注：

1. 虽然技术格式统一，但不同角色的使用条款可能完全不同。
2. 对站点来说，优先找能提供完整 runtime folder 的模型，而不是只有预览视频或编辑工程图。

#### B. 旧版 Live2D Cubism 2.1 模型

典型文件结构：

1. `*.model.json`
2. `*.moc`
3. `*.motion.json`
4. `*.physics.json`
5. 贴图资源

这类模型依然能用，因为 `pixi-live2d-display` 支持 Cubism 2.1 和 Cubism 4 两条 runtime 线。

主要获取侧：

1. 老版 Live2D sample
2. `pixi-live2d-display` 相关示例
3. 一些旧项目仓库或二次整理仓库

耳熟能详示例：

1. `Shizuku`

风险备注：

1. 老模型资料多、上手快，但画风和动作系统通常偏旧。
2. 新站如果后续要统一体验，建议把 Cubism 2.1 只作为验证或兼容测试，不作为最终主角色。

#### C. Spine / 游戏角色小人模型

典型文件结构：

1. `*.json` 或 `*.skel`
2. `*.atlas`
3. 对应贴图

这类模型不能直接丢给 Live2D runtime，需要改走 Spine runtime。

常见获取侧：

1. 游戏 Wiki 的模型展示页
2. 游戏资源查看站
3. 游戏提取仓库

耳熟能详示例：

1. 明日方舟 PRTS 干员小人
2. 一些手游的战斗 chibi / 基建小人

风险备注：

1. 技术上虽然可运行，但和当前 Live2D 挂件方案是不同技术栈。
2. 这类资源最容易出现“看得到、跑得起来，但来源和再部署风险都很高”的情况。

#### D. 模型聚合仓库和资源包

常见表现形式：

1. GitHub 聚合仓库
2. 网盘整理包
3. 第三方 demo 站点资源目录

技术上常常能直接拿到模型文件，但来源高度混杂。

风险备注：

1. 很适合做技术摸底。
2. 不适合作为主站长期资产来源基线。

### 6.9 当前方案可直接使用的模型获取面

如果以“能直接给当前 `pixi-live2d-display` 方案用”为标准，优先级如下：

#### 第一层：官方 Sample Data

这是最干净、最适合作为技术原型来源的一层。

原因：

1. 官方页面直接给出用于嵌入的 runtime folder。
2. 文件结构完整，通常包含 `moc3 / model3.json / motion3.json / exp3.json / physics3.json / cdi3.json`。
3. 与当前 Web 运行时路线天然兼容。

当前可作为技术储备关注的知名角色：

1. `Haru`
2. `Haru (receptionist version)`
3. `Hiyori Momose`
4. `Niziiro Mao`
5. `Hatsune Miku`
6. `Unity-chan`
7. `Shizuku`

适用建议：

1. `Haru / Hiyori / Mao` 适合现代 Web 挂件原型。
2. `Shizuku` 适合验证旧格式兼容链路。
3. `Hatsune Miku / Unity-chan` 适合做知名角色技术样本，但应单独核对角色使用限制。

#### 第二层：示例仓库和 demo 资源

主要是：

1. `pixi-live2d-display` README 和 demo 对应模型
2. Live2D 官方 `CubismWebSamples`
3. 作者公开的演示资源目录

技术价值：

1. 能快速验证你的加载代码是否正确。
2. 适合照着现成示例调试 hit area、motion、拖拽、自动交互等功能。

已确认的关键点：

1. `pixi-live2d-display` README.zh 直接提到其示例模型使用 `Shizuku (Cubism 2.1)` 和 `Haru (Cubism 4)`。
2. 这意味着当前工程路线同时兼容 `model.json/moc` 和 `model3.json/moc3` 两类模型。

#### 第三层：商店和委托平台

主要是：

1. `nizima`
2. `BOOTH`
3. `VGen`

技术价值：

1. 现代 Live2D 模型数量最多的实际来源。
2. 真正要做长期角色时，这一层比公共 GitHub 更现实。

限制：

1. 不会像官方 sample 那样集中提供统一格式说明。
2. 需要逐个商品看是否交付完整 runtime folder。

#### 第四层：公开聚合仓库

这层在技术上常常很有用，但只建议当“勘探层”。

用途：

1. 看看常见模型文件目录长什么样。
2. 测试 loader 是否兼容一些边缘结构。
3. 估计不同 IP、不同年代模型的组织方式。

限制：

1. 不同仓库里混有官方 sample、原创角色、游戏提取资源和来源不明资源。
2. 文档里只应把它记为“技术观察源”，不要把它默认当成部署源。

### 6.10 `pixi-live2d-display` 兼容面总结

根据 README.zh 当前可见说明，`pixi-live2d-display` 使用 Cubism 2.1 和 Cubism 4 runtime，因此覆盖：

1. Cubism 2.1：旧格式 `model.json + moc`
2. Cubism 3 / 4 兼容模型：现代格式 `model3.json + moc3`

这意味着你后续找模型时，技术判断可以先看入口文件：

1. 如果给的是 `xxx.model3.json`，基本在当前主方案范围内。
2. 如果给的是 `xxx.model.json`，通常也还能接，但属于旧格式。
3. 如果给的是 `xxx.atlas + xxx.json/skel`，那大概率是 Spine，不在当前 Live2D 主线里。

### 6.11 明日方舟 / PRTS 的技术获取面

如果你的目标非常明确，就是想尽量接近“明日方舟小人”，那技术上当前已确认的信息是：

1. PRTS 角色页确实提供“干员模型”展示。
2. 角色页源码里存在 `spine-root` 容器和 `SPINEDATA` 标记。
3. 以阿米娅为例，其 `/spine` 子页面里直接能看到 `prefix: https://static.prts.wiki/spine/`，并附带诸如：
   - `char/char_002_amiya/char_002_amiya/char_002_amiya`
   - `char/char_002_amiya/back_char_002_amiya/char_002_amiya`
   - `char/char_002_amiya/build_char_002_amiya/build_char_002_amiya`
4. 这说明从“技术获取”角度，PRTS 已经把明日方舟小人的 Spine 资源路径暴露得相当直接。

这部分信息带来的结论是：

1. 如果只说“哪里能拿到技术上可用的模型数据”，明日方舟小人的可观察来源是 `PRTS / static.prts.wiki/spine/` 这一系。
2. 但这条线拿到的是 Spine 资源，不是 Live2D 资源。
3. 所以后续如果你执意追求“明日方舟味的小人”，更现实的技术路线是：
   - 继续坚持 Live2D，但做原创仿气质角色
   - 或者另开一条 Spine 挂件路线

### 6.12 技术调研结论

如果单看“模型大多在哪边获取”，当前最清晰的地图如下：

1. 现代、结构完整、最适合当前方案的 Live2D 模型，主要在 `Live2D 官方 Sample Data`、`nizima`、`BOOTH`、作者公开发布页。
2. `pixi-live2d-display` 对接最顺的，是 `model3.json + moc3` 的现代 Cubism 模型。
3. `Shizuku` 这类旧 `model.json + moc` 模型仍可用于验证兼容链路。
4. 耳熟能详且技术上最容易直接拿到的官方样例角色，主要包括 `Haru`、`Hiyori Momose`、`Niziiro Mao`、`Hatsune Miku`、`Unity-chan`、`Shizuku`。
5. 明日方舟小人技术上主要可从 `PRTS / static.prts.wiki/spine/` 这一路观察到，但它属于 Spine，不属于 Live2D。
6. 因此“想要明日方舟小人”和“想要 Live2D 看板娘”在技术层面并不是同一个问题。

1. 默认桌面端启用
2. 移动端默认隐藏
3. 可通过配置禁用指定页面
4. 文章详情页保持较低视觉权重

### 7.3 渲染层与控制层分离

架构建议：

1. Vue 组件负责 UI、状态切换、用户事件。
2. 独立 engine 模块负责模型加载、参数更新、动作调用。
3. 气泡文案、冷却时间、是否显示等由状态模块管理。

## 8. 详细交互规范

### 8.1 默认位置与尺寸

桌面端建议：

1. 距右边 `20px` 到 `28px`
2. 距底边 `20px` 到 `28px`
3. 展开态宽度约 `180px` 到 `240px`
4. 高度随模型比例自适应

移动端建议：

1. 默认不展示完整模型
2. 如需保留，仅显示一个可展开入口

### 8.2 动效规范

1. 展开与收起时间控制在 `180ms` 到 `240ms`
2. 不做弹跳 overshoot
3. 不使用高频晃动
4. 视线跟随每帧要做插值和平滑，避免机械抖动

### 8.3 气泡文案规范

文案来源分三类：

1. 首次访问欢迎
2. 页面感知提示，例如进入博客页、联系页
3. 点击反馈短句

规范：

1. 单条 8 到 24 个汉字为宜
2. 不自动连续刷出
3. 每类提示应设置冷却

### 8.4 页面感知建议

可感知但不强打断的页面事件：

1. 进入首页时问候
2. 进入文章页时提示“正文模式”
3. 进入联系页时强调联系入口
4. 进入旧站归档页时保持低活跃，避免和历史归档主题冲突

## 9. 性能与稳定性要求

### 9.1 性能要求

1. 首屏不阻塞主要内容渲染。
2. 挂件脚本延后加载。
3. 模型资源懒加载，首次交互或空闲时再初始化。
4. 页面隐藏时暂停高频更新。

### 9.2 兼容性要求

1. 无 WebGL 时优雅降级为静态入口或完全禁用。
2. 用户设置 `prefers-reduced-motion` 时降低 motion 强度。
3. 移动端触屏下不做鼠标追踪逻辑。

### 9.3 可观测性

建议记录以下埋点或日志事件：

1. 模型加载成功/失败
2. 用户展开/关闭次数
3. 点击互动次数
4. 资源加载耗时

## 10. 授权与风险

### 10.1 Live2D 授权风险

Live2D 官方授权需要在正式落地前再次核对。尤其是：

1. Web 场景授权
2. 商业或公开站点使用边界
3. 若未来接 AI 对话，授权判断会更复杂

结论：

在模型资产和站点用途未最终确认前，不应直接把第三方模型接入生产站。

### 10.2 产品风险

1. 角色过于抢戏会压过博客内容。
2. 气泡过多会影响阅读。
3. 模型资源过大可能拖慢首屏。
4. 未经授权的角色素材会带来版权风险。

### 10.3 明日方舟模型风险结论

针对“用 PRTS 上能看到的明日方舟小人模型”这一想法，当前结论如下：

1. 技术上，PRTS 页面里的“小人模型”更像 Spine 展示，不是 Live2D `.moc3` 模型，和本方案的 Live2D runtime 不直接兼容。
2. 授权上，公开展示页面能看到模型，不等于允许把资源抽取后放进自己的公开网站。
3. 非商用、纯玩可以作为个人本地实验的动机，但不应作为公开部署的授权依据。
4. 生产规划里应把明日方舟模型列为“不进入主站”的路线。

## 12. 验收标准

1. 桌面端右下角悬浮稳定，不遮挡主导航和正文主要操作。
2. 鼠标移动时眼球跟随自然，头部仅轻微跟随。
3. 点击动作响应明确，但不会连续刷屏。
4. 用户可以轻松关闭或收起，并记住状态。
5. `npm run build` 通过。
6. 移动端不出现遮挡和误触问题。

## 13. 当前建议决策

如果后续正式开工，建议直接按下面的决策执行：

1. 第一阶段只做“装饰型助手”，不做 AI 对话。
2. 技术选型优先 `pixi-live2d-display` 路线。
3. 先用授权清楚的占位模型验证，再决定最终角色形象。
4. 最终角色优先走原创或委托，不使用明日方舟提取模型进入主站。
5. 待 UI 深色化方案稳定后再接入挂件，避免风格打架。
