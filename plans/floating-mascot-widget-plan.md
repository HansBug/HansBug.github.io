# 右下角悬浮看板娘挂件技术规划

更新日期：2026-04-10

## 1. 目标

为站点规划一个右下角悬浮的二次元角色挂件，支持待机、眨眼、呼吸、鼠标视线跟随、点击互动和可关闭/折叠，不嵌入正文流，不影响阅读主路径。

本规划优先考虑：

1. 可维护性
2. 页面性能
3. 交互克制
4. 资产和授权可控

## 2. 产品定义

### 2.1 定位

角色挂件不是首页主内容，也不是 AI 聊天产品。它的定位更像：

`全站级悬浮站点助手 / 值班记录员`

### 2.2 必须满足的交互前提

1. 固定在桌面端右下角悬浮。
2. 不进入正文布局，不挤占文章内容宽度。
3. 鼠标移动时，角色眼球必须能跟随指针。
4. 角色头部可以轻微跟随，但幅度要受控。
5. 支持点击头部或身体触发动作或短提示。
6. 用户可以收起、静音或关闭。
7. 移动端默认折叠或禁用，避免遮挡。

### 2.3 明确不做的内容

第一阶段不做：

1. AI 对话
2. 自动语音播报
3. 长时间占据屏幕的大气泡
4. 强打断式通知
5. 页面首次加载时的大型入场动画

## 3. 技术结论

### 3.1 推荐技术路线

推荐采用：

`Live2D 模型 + Cubism SDK for Web 能力模型 + PixiJS 渲染封装`

更具体地说：

1. 资产层：Live2D Cubism 模型
2. 运行时：Live2D Cubism SDK for Web
3. 集成层：优先评估 `pixi-live2d-display`
4. 前端封装：Astro 页面挂载 + Vue 交互控制组件

### 3.2 为什么不优先选 3D VRM

虽然 `three-vrm` 技术成熟，但不适合本阶段：

1. 运行时更重。
2. 对博客场景来说存在感太强。
3. 更接近 VTuber 头像，而不是轻量悬浮助手。
4. 角色风格更容易偏离当前站点的轻工业气质。

### 3.3 为什么不建议直接用“老式看板娘一键脚本”

这类脚本适合快速原型，不适合长期主方案：

1. 可维护性差。
2. 交互和资源管理粒度有限。
3. 很多项目维护状态不稳定。
4. 样式和生命周期很难与当前 Astro 站统一。

## 4. 方案对比

### 4.1 方案 A：官方 Live2D SDK 直接集成

优点：

1. 能力最完整。
2. 可精确控制参数、动作、命中区域和事件。
3. 后续升级空间最大。

缺点：

1. 集成成本更高。
2. 初次接入时工程细节较多。

适用场景：

1. 后续要做较复杂动作编排。
2. 计划长期维护角色系统。

### 4.2 方案 B：`pixi-live2d-display` 封装集成

优点：

1. 对 Web 集成更友好。
2. 常见交互如自动跟随、点击测试、动作调用更省事。
3. 更适合博客里这种局部挂件场景。

缺点：

1. 仍然依赖模型质量。
2. 对部分高级能力仍需理解底层参数。

适用场景：

1. 第一阶段就要尽快落地一个稳定挂件。
2. 后续仍然希望保留可扩展能力。

### 4.3 方案 C：第三方现成 widget 项目

优点：

1. 上线快。
2. 演示成本低。

缺点：

1. 样式难统一。
2. 可控性一般。
3. 长期维护风险高。

适用场景：

1. 只做一次性原型演示。

### 4.4 最终建议

首选 `方案 B`，保留未来必要时向 `方案 A` 下沉的空间。

## 5. 关键能力拆解

### 5.1 视线跟随

这是你明确要求的核心能力，通常依赖 Live2D 标准参数：

1. `ParamEyeBallX`
2. `ParamEyeBallY`
3. `ParamAngleX`
4. `ParamAngleY`
5. 必要时加入轻微 `ParamBodyAngleX`

实现原则：

1. 眼球跟随幅度可以稍大。
2. 头部跟随幅度必须小于眼球。
3. 身体跟随只做极弱补偿。
4. 鼠标离开窗口后平滑回正。

### 5.2 待机动画

建议配置：

1. 呼吸
2. 眨眼
3. 轻微头部浮动
4. 空闲 20 到 40 秒后触发一个低干扰 idle motion

### 5.3 点击互动

建议只保留有限命中区域：

1. 头部
2. 身体
3. 控制按钮区域

触发内容：

1. 播放一段短 motion
2. 显示一条短气泡文本
3. 记录冷却时间，避免连续刷屏

### 5.4 折叠与关闭

需要三种状态：

1. 展开
2. 收起为小按钮或头像徽标
3. 本次访问关闭

状态建议落在 `localStorage` 中持久化。

## 6. 资产要求

### 6.1 模型要求

要实现自然的“看着鼠标”，模型至少要满足：

1. 有完整眼球参数
2. 有头部角度参数
3. 最好有呼吸参数
4. 最好有基础表情和 2 到 4 个 motion
5. 有明确 hit area 配置

### 6.2 资产来源原则

必须满足以下要求：

1. 角色授权允许上站公开展示。
2. 运行时授权允许 Web 集成。
3. 不直接使用明日方舟或其他现成 IP 角色。
4. 最稳的是原创角色或明确可商用授权素材。

### 6.3 建议角色设定

为了与博客气质匹配，角色形象建议是：

1. 偏记录员、技术助理、档案管理员
2. 服装和配色低饱和
3. 不要过度萌系、亮色、糖果感
4. 表情偏冷静、温和、轻陪伴

## 7. 前端集成架构

### 7.1 推荐文件落点

建议新增：

1. `src/components/FloatingMascotWidget.vue`
2. `src/components/FloatingMascotBubble.vue`
3. `src/utils/mascot/engine.ts`
4. `src/utils/mascot/state.ts`
5. `public/mascot/` 下的模型与贴图资源

建议修改：

1. `src/layouts/BaseLayout.astro`
2. `src/styles/global.css`
3. `src/config/site.ts` 或新增 `src/config/mascot.ts`

### 7.2 挂载方式

推荐在 `BaseLayout.astro` 中全站挂载，但加条件控制：

1. 默认桌面端启用
2. 移动端默认隐藏
3. 可通过配置禁用指定页面
4. 文章详情页保持较低视觉权重

### 7.3 渲染层与控制层分离

架构建议：

1. Vue 组件负责 UI、状态切换、用户事件。
2. 独立 engine 模块负责模型加载、参数更新、动作调用。
3. 气泡文案、冷却时间、是否显示等由状态模块管理。

## 8. 详细交互规范

### 8.1 默认位置与尺寸

桌面端建议：

1. 距右边 `20px` 到 `28px`
2. 距底边 `20px` 到 `28px`
3. 展开态宽度约 `180px` 到 `240px`
4. 高度随模型比例自适应

移动端建议：

1. 默认不展示完整模型
2. 如需保留，仅显示一个可展开入口

### 8.2 动效规范

1. 展开与收起时间控制在 `180ms` 到 `240ms`
2. 不做弹跳 overshoot
3. 不使用高频晃动
4. 视线跟随每帧要做插值和平滑，避免机械抖动

### 8.3 气泡文案规范

文案来源分三类：

1. 首次访问欢迎
2. 页面感知提示，例如进入博客页、联系页
3. 点击反馈短句

规范：

1. 单条 8 到 24 个汉字为宜
2. 不自动连续刷出
3. 每类提示应设置冷却

### 8.4 页面感知建议

可感知但不强打断的页面事件：

1. 进入首页时问候
2. 进入文章页时提示“正文模式”
3. 进入联系页时强调联系入口
4. 进入旧站归档页时保持低活跃，避免和历史归档主题冲突

## 9. 性能与稳定性要求

### 9.1 性能要求

1. 首屏不阻塞主要内容渲染。
2. 挂件脚本延后加载。
3. 模型资源懒加载，首次交互或空闲时再初始化。
4. 页面隐藏时暂停高频更新。

### 9.2 兼容性要求

1. 无 WebGL 时优雅降级为静态入口或完全禁用。
2. 用户设置 `prefers-reduced-motion` 时降低 motion 强度。
3. 移动端触屏下不做鼠标追踪逻辑。

### 9.3 可观测性

建议记录以下埋点或日志事件：

1. 模型加载成功/失败
2. 用户展开/关闭次数
3. 点击互动次数
4. 资源加载耗时

## 10. 授权与风险

### 10.1 Live2D 授权风险

Live2D 官方授权需要在正式落地前再次核对。尤其是：

1. Web 场景授权
2. 商业或公开站点使用边界
3. 若未来接 AI 对话，授权判断会更复杂

结论：

在模型资产和站点用途未最终确认前，不应直接把第三方模型接入生产站。

### 10.2 产品风险

1. 角色过于抢戏会压过博客内容。
2. 气泡过多会影响阅读。
3. 模型资源过大可能拖慢首屏。
4. 未经授权的角色素材会带来版权风险。

## 11. 分阶段执行计划

### Phase 1：技术预研与资产确认

目标：

1. 确认角色来源和授权。
2. 用占位模型验证渲染、跟随和点击链路。

产出：

1. 可运行的本地 demo
2. 视线跟随平滑参数
3. 可关闭和折叠机制

### Phase 2：站点集成

目标：

1. 接入 `BaseLayout.astro`
2. 实现右下角浮层和基本气泡
3. 完成桌面端稳定展示

产出：

1. 全站可复用挂件组件
2. 页面级开关配置
3. 基础页面感知提示

### Phase 3：体验打磨

目标：

1. 调整尺寸、层级、冷却和文案。
2. 处理移动端和低性能设备降级。

产出：

1. 完整交互细节
2. 性能优化
3. 生产可发布版本

## 12. 验收标准

1. 桌面端右下角悬浮稳定，不遮挡主导航和正文主要操作。
2. 鼠标移动时眼球跟随自然，头部仅轻微跟随。
3. 点击动作响应明确，但不会连续刷屏。
4. 用户可以轻松关闭或收起，并记住状态。
5. `npm run build` 通过。
6. 移动端不出现遮挡和误触问题。

## 13. 当前建议决策

如果后续正式开工，建议直接按下面的决策执行：

1. 第一阶段只做“装饰型助手”，不做 AI 对话。
2. 技术选型优先 `pixi-live2d-display` 路线。
3. 先用占位模型验证，再决定最终角色形象。
4. 待 UI 深色化方案稳定后再接入挂件，避免风格打架。
