## 工具链越多，不一定越专业

对很多个人项目和中小型内部项目来说，真正需要的通常不是十几种工具，而是一套**边界清楚、职责不重叠**的最小组合。

## 我更偏好的组合方式

### 包管理和依赖锁定

先把依赖装清楚、版本锁清楚。没有这一层，后面谈可复现基本都站不住。

### 代码质量工具

格式化、lint、类型检查至少选一套能持续跑的，不要求一上来全满配，但要能自动执行。

### 入口与目录结构

尽量早点从“单脚本”转成“有明确入口和模块边界”的结构。不是为了好看，而是为了以后不至于全靠搜索函数名找逻辑。

## 从脚本到项目的真正转折点

真正的转折通常不是文件数变多，而是你开始思考：

- 哪些部分是复用逻辑
- 哪些部分是配置
- 哪些部分是实验代码
- 哪些部分应该被测试保护

一旦开始分这些边界，你就已经在从“写脚本”走向“做项目”了。
