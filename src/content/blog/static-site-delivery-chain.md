## 发布链路的目标不是炫技

静态站点的 CI/CD 不需要很复杂。对个人博客来说，最关键的是两件事：

1. 本地构建结果和线上构建结果尽量一致。
2. 发布步骤不要依赖手工点击。

一旦发布还需要靠人去页面里点几次，你就已经把维护成本留到未来了。

## 一个够稳的最小流程

我通常建议把流程收敛成：

### checkout

拉取代码和内容。

### install

固定依赖安装方式，避免每次环境都不一样。

### build

做静态构建，同时把类型检查和基本校验放进来。

### deploy

把 `dist` 或 `public` 上传到 GitHub Pages。

这个流程不追求复杂功能，而追求**任何时候都能重新跑一遍且结果稳定**。

## 发布链路里最常见的问题

一个是路径问题。GitHub 用户站点和项目站点的 base 路径不一样，很多站点就是卡在这里。

一个是权限问题。Pages 工作流需要 `pages: write` 和 `id-token: write`，少一个都可能卡住。

还有一个是“明明本地能跑，线上构建挂了”。这通常不是玄学，而是因为 Node 版本、锁文件或构建命令不一致。

## 为什么我会把它归到调试排障

因为部署本身也是一条系统链路。只要链路里有多个环节，就会有观测、定位和修复的问题。把部署看成一个需要长期稳定运行的系统，比把它看成“最后顺手点一下”更靠谱。
